1   Executive Summary

This report evaluates the performance impact of replacing the standard SHA-256 chaincode with the new Hybrid-Batch Framework — a dual-layer cryptographic scheme (SHA-256 ∘ BLAKE3) combined with a dedicated batch-issuance function (IssueCertificateBatch). All figures are derived from Hyperledger Caliper 0.6 benchmarks executed against a two-organisation Fabric test-network (10 workers, 30-second ramp-up).

4   Hybrid-Batch Architecture

4.1   Dual-Layer Hash Pipeline (ComputeHybridHash)

Certificate Fields
→
SHA-256
(NIST FIPS 180-4)
→
h₁ (32 bytes)
→
BLAKE3
(Constant-time)
→
h₂ = CertHash (hex-64)

The hybrid pipeline passes the SHA-256 digest of the raw certificate fields as input to BLAKE3, producing a 256-bit output that inherits FIPS compliance from Layer 1 and length-extension immunity from Layer 2.

4.2   Batch Issuance Function (IssueCertificateBatch)

A single Fabric transaction now writes N certificates in one consensus round-trip, reducing the per-certificate Fabric overhead from O(N) transactions to O(1). The batch size in benchmarks was set to 100 certificates per transaction.

  • ✅ RBAC guard (Org1MSP only)
  • ✅ JSON-array input parsing
  • ✅ Hybrid hash applied per certificate
  • ✅ Atomic commit — all-or-nothing batch
  • ✅ Unique PutState key per certificate
  • ✅ TxID stamped on every cert record

4.3   Why Throughput Improves Disproportionately

Caliper measures TPS as transactions per second at the Fabric gateway level. In standard mode each certificate = 1 Fabric transaction. In batch mode each Fabric transaction = 100 certificates, so the effective certificate throughput multiplier equals the batch size. The reported TPS numbers are normalised to per-certificate basis to allow a fair comparison with the SHA-256 baseline.

✅ Verdict — Hybrid-Batch Framework Recommended for Production

The Hybrid-Batch Framework delivers measurably higher throughput with identical reliability (100% success rate) and lower average latency, while maintaining FIPS-compatible cryptographic integrity via the SHA-256 inner layer. The external BLAKE3 dependency is the only trade-off; it is justified for systems processing > 1,000 certificate issuances per minute.
