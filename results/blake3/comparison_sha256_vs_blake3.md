## Research Interpretation

### Why BLAKE3 Outperforms SHA-256 in Blockchain Workloads

**Hash Function Properties:**

| Property | SHA-256 | BLAKE3 |
|----------|---------|--------|
| Output size | 256 bits | 256 bits |
| Security level | 128-bit collision | 128-bit collision |
| Algorithm type | Merkle–Damgård | Tree-based Merkle |
| SIMD support | Partial (x86) | AVX-512, AVX2, NEON |
| Throughput (software) | ~250–350 MB/s | ~800–3000 MB/s |
| Go library | `crypto/sha256` | `github.com/zeebo/blake3` |
| Client npm | `crypto` (built-in) | `blake3` npm package |

**Performance Impact on Hyperledger Fabric:**

1. **Endorsement phase** — Peer recomputes `BLAKE3(fields)` per IssueCertificate call.
   Faster hashing → lower CPU per endorsement → higher endorsement TPS ceiling.
2. **Validation phase** — Orderer/committer validates endorsed blocks;
   BLAKE3's parallelism benefits multi-certificate block validation.
3. **Read path** — VerifyCertificate recomputes hash client-side for comparison.
   BLAKE3's speed reduces client overhead especially under high read TPS.
4. **On-chain integrity** — `HashAlgorithm: "BLAKE3"` field ensures every
   certificate record explicitly declares its hash algorithm, enabling
   future algorithm rotation without breaking historical verification.
